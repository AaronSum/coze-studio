# Copilot Instructions – @coze-studio/studio-i18n-resource-adapter

> 子包路径：frontend/packages/arch/resources/studio-i18n-resource
>
> 包名：@coze-studio/studio-i18n-resource-adapter
>
> 角色定位：**纯数据 & 类型定义的 i18n 资源适配层**，为 Coze Studio 前端应用提供多语言文案和类型安全。

---

## 1. 全局架构与职责边界

- **单一职责**：本包只负责
  - 提供多语言文案数据（目前：`en`、`zh-CN`）；
  - 提供与文案 key / 参数相关的类型定义；
  - 暴露一个适合 i18n 库（如 i18next）的 `defaultConfig` 结构。
- **核心入口**：
  - [src/index.ts](src/index.ts)
    - `import localeEn from './locales/en.json'`
    - `import localeZhCN from './locales/zh-CN.json'`
    - 构造
      - `defaultConfig = { en: { i18n: localeEn }, 'zh-CN': { i18n: localeZhCN } }`；
      - `export { localeEn, localeZhCN, defaultConfig }`；
      - 从 [src/locale-data.d.ts](src/locale-data.d.ts) re-export 类型（`I18nOptionsMap`、`I18nKeysHasOptionsType` 等）。
- **资源来源**：
  - [README.md](README.md) 指出：所有 locale JSON 和类型文件 **通过脚本自动生成**（`dl-i18n` 命令），不建议手写修改。
- **数据流**：
  - 上游：内部 i18n 平台 / 翻译源（通过 `pnpm dl-i18n` 拉取并生成 `en.json` / `zh-CN.json` 和类型定义）；
  - 本包：组装、导出 locale 资源和 TS 类型；
  - 下游：Studio 前端应用直接 `import { defaultConfig, localeEn, localeZhCN }` 来初始化 i18n 实例或访问文案。
- **架构意图**：
  - 将「文本内容 & 语言扩展 & TS 类型」集中在一个 **独立、可缓存的 workspace 包** 中；
  - 避免在业务包重复维护 keys/type；
  - 为 React / i18next 场景提供现成、强类型的资源对象结构。

## 2. 关键开发工作流（build / lint / test / 资源生成）

### 2.1 包级脚本

见 [package.json](package.json)：

- `build`: `exit 0`
  - **目前是空实现**，不做实际构建；构建一般由上层 Rush/monorepo 流程驱动。
- `lint`: `eslint ./ --cache`
  - 使用仓库统一的 `@coze-arch/eslint-config`（见 [eslint.config.js](eslint.config.js)）。
- `test`: `exit 0`
- `test:cov`: `exit 0`
  - Vitest 配置在 [vitest.config.ts](vitest.config.ts)，但当前包级 test 脚本未真正调用；
  - 若要在本包内跑测试，可在 monorepo 级配置用 rushx 调用 vitest，或本地临时执行（例如在根用 `vitest --config frontend/packages/arch/resources/studio-i18n-resource/vitest.config.ts`）。

### 2.2 TypeScript 构建配置

- 顶层 TS 配置：[tsconfig.json](tsconfig.json)
  - 仅声明 `"composite": true`，并通过 `references` 引用：
    - `./tsconfig.build.json`
    - `./tsconfig.misc.json`
  - `exclude: ["**/*"]`：顶层 tsconfig 本身不直接参与编译，仅作为项目引用入口，遵循 monorepo TS Project References 结构。
- 构建用 TS 配置：[tsconfig.build.json](tsconfig.build.json)
  - `extends: "@coze-arch/ts-config/tsconfig.node.json"`：继承统一的 Node/打包规范；
  - `compilerOptions` 关键项：
    - `jsx: "preserve"`：允许 React/JSX 类型使用；
    - `rootDir: "./src"`；
    - `outDir: "./dist"`；
    - `tsBuildInfoFile: "dist/tsconfig.build.tsbuildinfo"`；
  - `include: ["src", "src/**/*.json"]`：明确将 JSON 资源纳入 TS 构建；
  - `references` 指向 monorepo 统一配置包（eslint-config / ts-config / vitest-config）。

### 2.3 测试配置（Vitest）

- [vitest.config.ts](vitest.config.ts)
  - 使用 `@coze-arch/vitest-config` 的 `defineConfig`：
    - `dirname: __dirname`；
    - `preset: 'web'`；
    - `coverage.all = true`；
    - `coverage.exclude = ['starling.config.js', 'src/resource', 'script/dl-i18n.js']`。
  - 特点：
    - 通过 monorepo 封装的 preset 统一 Web 端测试行为；
    - 显式排除生成脚本与资源目录，避免将大体量 i18n 数据纳入覆盖率基数。

### 2.4 资源生成 / 更新

- 文案和类型为 **自动生成**，核心流程见 [README.md](README.md)：
  - 使用命令：`pnpm dl-i18n`（在 monorepo 顶层或相关脚本位置执行）；
  - 过程会：
    - 下载/同步最新翻译；
    - 更新 `src/locales/en.json` 与 `src/locales/zh-CN.json`；
    - 更新 `src/locale-data.d.ts` 中的类型定义；
    - 保证 [src/index.ts](src/index.ts) 的导出结构与生成结果保持一致（通常也是生成的一部分）。
- **注意**：不要手动改这些生成产物（`en.json`、`zh-CN.json`、`locale-data.d.ts`、自动块标记为“Automatically generated by the dl-i18n command”的文件），修改会在下次 `dl-i18n` 执行时被覆盖。

## 3. 项目特有约定与模式

### 3.1 i18n 资源导出约定

- 统一使用 `defaultConfig` 作为 i18n 库初始化资源：
  - 结构：
    ```ts
    const defaultConfig = {
      en: { i18n: localeEn },
      'zh-CN': { i18n: localeZhCN },
    } as {
      en: { i18n: typeof localeEn };
      'zh-CN': { i18n: typeof localeZhCN };
    };
    ```
  - 下游代码一般：
    - `resources: defaultConfig`；
    - `defaultNS: 'i18n'`；
    - `config.en.i18n` / `config['zh-CN'].i18n` 访问对应语言。
- 直接 locale object 访问：
  - `localeEn.account_login_success` 之类，适合简单场景或无需 i18n 实例时快速取文案。

### 3.2 类型安全的文案 key / 参数

- 类型定义文件：[src/locale-data.d.ts](src/locale-data.d.ts)
  - 暴露的核心类型（通过 [src/index.ts](src/index.ts) re-export）：
    - `I18nKeysHasOptionsType`: 所有 **需要参数插值** 的 key 的 union；
    - `I18nKeysNoOptionsType`: 所有 **不需要参数** 的 key 的 union；
    - `I18nOptionsMap`: 从 key 映射到参数对象类型，如：
      - `I18nOptionsMap['AddSuccessToast'] // => { name: ReactNode }`；
    - `LocaleData`: locale 数据结构定义。
- 推荐使用模式（例如 i18next + React）：
  - 
    ```ts
    import { type I18nOptionsMap } from '@coze-studio/studio-i18n-resource-adapter';
    import { useTranslation } from 'react-i18next';

    const { t } = useTranslation();

    t('AddSuccessToast', {
      name: 'My Bot',
    } satisfies I18nOptionsMap['AddSuccessToast']);
    ```
  - 这样可以在调用层确保参数完整和类型正确，符合本项目的「强类型文案参数」约定。

### 3.3 Lint / TS / Test 的统一 preset

- Lint：
  - [eslint.config.js](eslint.config.js) 使用 `@coze-arch/eslint-config` + `preset: 'web'`；
  - 包内不自定义额外规则（`rules: {}`），保持与 monorepo 统一。
- TS：
  - 所有 TS 选项优先继承 `@coze-arch/ts-config`；
  - 本包只补充与输出目录、build info 相关的选项，尽量不在子包本身写杂项配置。
- Test：
  - `@coze-arch/vitest-config` 封装通用 vitest 规则；
  - 子包只指定 preset 和覆盖率过滤，避免与 monorepo 其他包行为不一致。

### 3.4 只读数据包的开发约束

- 本包无 runtime 依赖，仅有：
  - `peerDependencies`: `react`, `react-dom`（保证 `ReactNode` 类型存在）。
- 通常不在此包写业务逻辑或组件，只维护：
  - 数据（JSON）、
  - TS 类型、
  - 极薄的一层导出 glue code（见 [src/index.ts](src/index.ts)）。
- 若需要新增逻辑工具函数（例如包装 t 函数）：
  - 优先放到其他业务/infra 包，不要把非「资源/类型」逻辑堆进这个 adapter。

## 4. 与外部/其他子包的集成细节

### 4.1 与 i18n 库（如 i18next / react-i18next）

- 典型接入方式（简化自 [README.md](README.md)）：

  ```ts
  import { defaultConfig } from '@coze-studio/studio-i18n-resource-adapter';
  import i18n from 'i18next';

  i18n.init({
    resources: defaultConfig,
    lng: 'en',
    fallbackLng: 'en',
    defaultNS: 'i18n',
    interpolation: { escapeValue: false },
  });
  ```

- 注意：
  - `defaultConfig` 的结构已经符合 i18next `resources` 预期，**不要再额外包一层 namespace**；
  - 若新增语言，需要按照 README 的步骤扩展 `src/locales` 和 `defaultConfig`。

### 4.2 与 React 集成

- 通过类型依赖提供 `ReactNode` 支持：
  - 某些文案 key 的参数类型为 `ReactNode`；
  - 下游组件在传参时可以安全传入 React 元素。
- 由于本包本身不直接 import React，而是只依赖类型：
  - 使用 `peerDependencies`（`react`, `react-dom`）来避免重复安装；
  - 确保由应用侧提供实际 React 运行时。

### 4.3 与 monorepo 配置包

- DevDeps 中的统一配置：
  - `@coze-arch/eslint-config`
  - `@coze-arch/ts-config`
  - `@coze-arch/vitest-config`
- TS `references` 引用这些配置包的 tsconfig：
  - 确保在 Rush/TS Project References 构建链路中，本包可以正确被依赖和增量编译。

## 6. 该子包的特殊点 & 使用注意事项

- **1）纯数据包 + 自动生成**
  - 几乎所有核心内容来自生成脚本，即使 `src/index.ts` 也带有「Automatically generated」标识；
  - AI/开发者在修改前，需要确认是否会被生成流程覆盖，避免做无效改动。

- **2）强类型 i18n key 模式**
  - 与常见“自由字符串 key”不同，本包对 key 和参数建立了完整的 TS 类型映射：
    - 方便 IDE 自动补全；
    - 在 refactor / 删除 key 时可以通过类型错误快速发现受影响调用点。

- **3）JSON 资源直接参与 TS 编译**
  - `tsconfig.build.json` 中 `include: ["src", "src/**/*.json"]`，说明 JSON 将通过 TS loader 参与构建；
  - 在新增 locale 文件时，应放在 `src/locales` 下并保证路径/命名与生成脚本一致。

- **4）软禁直接 build/test**
  - `build` / `test` 脚本目前是 `exit 0`，意味着单独在此包运行 `pnpm build` / `pnpm test` 不会做任何事；
  - 实际构建/测试一般通过 **上层 monorepo 命令** 统一触发。

- **5）规模较大的文案集**
  - README 指出翻译条目数量达 13k+；
  - 在写工具或测试时，避免对整个 JSON 做全量遍历/快照断言，以免拖慢 CI；
  - 建议针对具体 key、具体类型做小而精的单测。

## 7. 对 AI Coding Agent 的具体建议

- 在本子包中编写/修改代码时：
  - **优先保持「只做导出和类型 glue」的角色**，不要引入复杂逻辑或运行时代码；
  - 如需扩展语言：**同时修改** `src/locales/*.json`（通常通过 `dl-i18n`）、`src/index.ts`、`LocaleData` 类型，而非只改其中一处；
  - 遵循 monorepo 统一配置（eslint / tsconfig / vitest-config），不要在子包私自建立一套新的代码风格或编译方式。
- 若需要在其他包里接入 i18n：
  - 优先导入 `defaultConfig` 作为 i18n 的 `resources`；
  - 使用 `I18nOptionsMap` / `I18nKeysHasOptionsType` 带上 `satisfies` 来保证参数类型安全；
  - 不要在下游项目复制/粘贴本包的 JSON/类型，而是作为依赖引用。
